# ATLAS — AI Safety Design Notes

This document describes the safety and reliability decisions built into ATLAS. It is intended to demonstrate how safety properties are designed into an AI assistant from the ground up, not bolted on afterward.

---

## 1. Bounded Memory & Context Windows

**Risk:** Unbounded conversation history enables prompt injection via old turns — an attacker (or confused user) could plant instructions early in the conversation that influence later LLM calls.

**Mitigations:**
- `ConversationMemory` enforces a hard cap of 100 items (`max_items`). When the cap is hit, the oldest turns are dropped via `items[-max_items:]`.
- The LLM context block passed to Claude is further limited to the 5 most **relevant** turns (`relevant(query, top_k=5)`), not a full dump of history.
- Memory is loaded from disk with a truncation guard: `data[-self.max_items:]` prevents a manually inflated file from blowing the window.

```python
# memory.py
self.items = [str(x) for x in data][-self.max_items :]
```

---

## 2. Secret Redaction Before Output

**Risk:** API keys, bearer tokens, and passwords could leak into logs, diagnostics, or user-visible responses.

**Mitigations:**
- All diagnostic output is passed through `redact_secrets(text)` before being returned to the user or logged.
- `security.py` maintains pre-compiled regex patterns (no per-call recompilation) for: Anthropic keys (`sk-ant-*`), generic API key assignments, `Authorization: Bearer` headers, and generic `password/secret/token` assignments.
- `mask_value()` is used when displaying key existence — shows only the first 4 characters.

```python
# assistant.py — diagnostics path
snapshot = build_diagnostics_snapshot(...)
text = redact_secrets(snapshot.to_text())   # always redacted before return
```

---

## 3. Graceful Degradation (Fallback Mode)

**Risk:** Hard dependency on the LLM API means the assistant becomes completely non-functional when the API is unavailable.

**Mitigations:**
- If no API key is found at startup, `AtlasEngine` operates in fallback mode — all commands except LLM queries still work (calendar, notes, voice, memory, etc.).
- If an LLM call raises an exception at runtime, the engine catches it, logs a warning, and returns a structured fallback string rather than propagating the error.
- The fallback message includes the user's input and current context so the user can still act on it.

```python
# engine.py
except Exception as exc:
    logger.warning("LLM call failed (%s: %s), using fallback.", type(exc).__name__, exc)
    return EngineResponse(text=self._fallback_reply(...), used_live_llm=False)
```

---

## 4. Input Size Limits (DoS Prevention)

**Risk:** Accepting unbounded user input enables memory exhaustion attacks — a single large message could consume all available RAM or disk.

**Mitigations:**
- Pydantic `Field(max_length=...)` enforces hard limits on all API request fields before they reach application logic:
  - Chat messages: 4,000 characters
  - Notes/titles: 2,000 characters
  - Email bodies: 8,000 characters
  - Smart home device/value fields: 64 characters
- Integration layer also applies `[:_MAX_*]` slicing before writing to disk, providing defence in depth.

```python
# api.py
class ChatRequest(BaseModel):
    message: str = Field(..., max_length=4_000)
```

---

## 5. Atomic File Writes (Data Integrity)

**Risk:** A crash or concurrent write mid-operation could corrupt the JSON data files, causing silent data loss.

**Mitigations:**
- All JSON writes use an atomic write pattern: data is written to a `.tmp_*.json` file first, then `Path.replace()` (atomic rename on POSIX, best-effort on Windows) swaps it in place.
- Per-file `threading.Lock` instances prevent concurrent writes to the same file from racing.
- On write failure the temp file is cleaned up and the error is logged — the original file is never partially overwritten.

```python
# integrations.py
tmp_path.replace(file_path)   # atomic rename
```

---

## 6. No Shell Execution from User Input

**Risk:** Passing user-controlled strings to shell commands enables command injection.

**Design decision:** ATLAS never passes user input to `subprocess`, `os.system`, `eval`, or `exec`. All data flows through Python data structures (dicts/lists → JSON). The only external process call in the voice path uses a pre-determined file path generated by `tempfile.mkstemp`, not user input.

---

## 7. Authentication & Authorisation

**Risk:** An unauthenticated API exposed on a local network allows any device to read/write personal data.

**Mitigations:**
- `ATLAS_API_AUTH_KEY` enables bearer-token authentication on all API endpoints. When set, requests without a valid `Authorization: Bearer <key>` header receive `401 Unauthorized`.
- Auth is **off by default** (empty key = open), making the assistant usable out of the box for local/trusted use without requiring setup.
- Auth state is checked on every request via FastAPI `Depends(_check_auth)` — no route bypasses it.

---

## 8. CORS Lockdown

**Risk:** A wildcard `Access-Control-Allow-Origin: *` allows any website to make authenticated cross-origin requests to the API.

**Mitigation:** `ATLAS_CORS_ORIGINS` allows operators to restrict CORS to specific origins (e.g., the Tailscale IP). The default (`*`) is permissive for development convenience; restricting it is a one-env-var change for production.

---

## 9. Error Containment — No Traceback Leakage

**Risk:** Unhandled exceptions can leak file paths, internal state, and dependency versions to API clients.

**Mitigation:** A global FastAPI exception handler catches all unhandled exceptions and returns a generic `{"detail": "Internal server error."}` — no Python traceback is ever sent to the client. The real error is available in server logs.

```python
# api.py
@app.exception_handler(Exception)
async def _unhandled(_req, exc):
    return JSONResponse(status_code=500, content={"detail": "Internal server error."})
```

---

## 10. Structured Command Routing (Minimising LLM Surface)

**Risk:** Routing all commands through the LLM expands the attack surface for prompt injection and increases the chance of hallucinated actions.

**Design decision:** ATLAS uses a whitelist of structured commands (`_is_structured()`) that are handled entirely in Python without any LLM call. The LLM is only invoked for free-form conversational queries that genuinely require generative capability. This means calendar, notes, email, smart home, diagnostics, and briefing commands are immune to prompt injection by design.

---

## Summary

| Safety Property | Mechanism | Location |
| --- | --- | --- |
| Bounded context | Hard `max_items` cap, top-k relevance filter | `memory.py` |
| Secret redaction | Pre-compiled regex, applied before all output | `security.py`, `assistant.py` |
| Graceful degradation | Fallback mode, per-call exception handling | `engine.py` |
| Input size limits | Pydantic `Field(max_length)` + slice guards | `api.py`, `integrations.py` |
| Data integrity | Atomic rename writes + per-file thread locks | `integrations.py` |
| No shell injection | No `subprocess`/`eval` on user input | all modules |
| API authentication | Bearer token via `ATLAS_API_AUTH_KEY` | `api.py` |
| CORS lockdown | Configurable via `ATLAS_CORS_ORIGINS` | `api.py` |
| Error containment | Global exception handler, no traceback leakage | `api.py` |
| Minimised LLM surface | Structured command whitelist | `assistant.py` |
